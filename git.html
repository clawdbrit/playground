Git Branches & Commits

How commits stay isolated, and what branching actually looks like.

The concept

A commit is a snapshot of your project at a moment in time — not just the changed file, the whole state. Git stores these as a chain: each commit knows which one came before it.

A branch is just a pointer to a commit. When you make a new commit, that pointer moves forward. Other branches don't move — they stay where they are. That's the whole thing.

You're always on exactly one branch. When you commit, only that branch advances. Use the buttons below to build out a repo and watch the graph update.

What the commands look like
# you're on main, just initialized
git init

# make a commit on main
git add .
git commit -m "first commit"

# branch off and do some work
git checkout -b feature-a
git commit -m "add feature"

# push to github
git push origin feature-a

# merge back into main when ready
git checkout main
git merge feature-a
Why it matters

The key insight: your local repo and GitHub are two separate copies. Until you push, GitHub has no idea what you've done. You can commit freely, experiment, even blow things up — it's all local. Push only when you're ready.

Branches let multiple things happen in parallel without interfering. main stays clean and working. You build in feature-a. When it's ready, you merge it back. If it's a disaster, you just delete the branch — main never knew.

Switching branches snaps your files. When you git checkout feature-b, the actual files on disk change to match that branch. It feels like magic. Git is swapping out your whole file system state in milliseconds.
